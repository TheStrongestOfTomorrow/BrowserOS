## How Can I Contribute?

### Reporting Bugs
This section guides you through submitting a bug report for browserOS. Following these guidelines helps maintainers and the community understand your report, reproduce the behavior, and find related reports.
- **Check for existing issues**: Before opening a new issue, please search the issue tracker to see if the bug has already been reported.
- **Provide a clear description**: Include details about your environment, the steps to reproduce the bug, and the expected vs. actual behavior.
- **Include screenshots/videos**: Visual aids are extremely helpful for debugging UI issues.

### Suggesting Enhancements
This section guides you through submitting an enhancement suggestion for browserOS, including completely new features and minor improvements to existing functionality.
- **Explain the "Why"**: Describe the problem you're trying to solve or the benefit your suggestion would bring.
- **Provide examples**: If possible, show how other projects handle similar features.

### Your First Code Contribution
Unsure where to begin contributing to browserOS? You can start by looking through `beginner` and `help-wanted` issues.
- **Fork the repository**: Create your own copy of the project to work on.
- **Create a branch**: Use a descriptive name for your branch (e.g., `feature/add-dark-mode`).
- **Submit a Pull Request**: Once your changes are ready, submit a PR for review.

## Styleguides

### Git Commit Messages
- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests liberally after the first line

### Documentation
- Use Markdown for all documentation.
- Keep documentation up-to-date with code changes.
- Use clear and concise language.

### Coding Standards
- Follow the existing code style and naming conventions.
- Write clean, maintainable, and well-documented code.
- Ensure your changes don't break existing functionality.

## Credits & Recognition

We value every contribution, whether it's a bug fix, a new feature, or a documentation update. All contributors will be listed in our `README.md` and recognized in our community.
